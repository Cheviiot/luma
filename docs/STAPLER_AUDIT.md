# Аудит соответствия Luma документации Stapler

Дата проверки: 2026-06-03.

## Использованные разделы документации

- [Введение](https://stplr.dev/docs/intro/): назначение Stapler, поддерживаемые сценарии и базовые команды пользователя.
- [Stapler-репозиторий](https://stplr.dev/docs/dev/repository/): структура многопакетного Git-репозитория, корневой `stapler-repo.toml`, совпадение имени каталога и `name` в `Staplerfile`.
- [Staplerfile](https://stplr.dev/docs/dev/staplerfile/): обязательные поля, архитектуры, лицензии, `provides`, `replaces`, `sources`, `checksums`, lifecycle-скрипты и distro overrides.
- [stplr-spec](https://stplr.dev/docs/dev/stplr-spec/): `.stapler/update-check`, автоматическое обновление версии/checksums и `clean-build` для изолированной сборки.

## Текущее состояние Luma

- Репозиторий использует многопакетную структуру: каждый пакет находится в отдельном каталоге с собственным `Staplerfile`.
- Корневой `stapler-repo.toml` содержит обязательные поля репозитория и совместим с `stplr >= v0.1.1`.
- Все пакеты имеют явные `provides` и `replaces`, что помогает Stapler и системному пакетному менеджеру сопоставлять короткие имена пакетов.
- Для каждого пакета есть `.stapler/update-check`, `LICENSE`, `postinstall.sh` и `postremove.sh`.
- README содержит витрину пакетов, команды установки, полную сводку версий, лицензий и архитектур.

## Что усилено

- Валидатор теперь разбирает строки в одинарных и двойных кавычках, поэтому проверяет реальные `sources`, а не только `checksums`.
- Проверяется корневой `stapler-repo.toml`: обязательные поля, формат `minVersion` и шаблон `report_url`.
- Проверяются локальные `local:///` источники: файл должен существовать рядом с пакетом.
- Проверяются `stapler-repo.toml` внутри пакетов: допускаются рабочие форматы `include = "../stapler-repo.toml"` и `[inherit]`.
- Проверяются `LICENSE`, исполняемый `.stapler/update-check` и исполняемые файлы из массива `scripts`.
- Проверяются обязательные пользовательские и русскоязычные поля metadata: `summary`, `summary_ru`, `desc`, `desc_ru`, `group`, `homepage`, `maintainer`.
- Нестандартные лицензии приведены к документированному значению `Custom`.

## Следующие шаги

- Постепенно нормализовать наследование `stapler-repo.toml` к одному стилю во всех пакетах.
- Добавить выборочную CI-сборку измененных пакетов, а не только ручной `workflow_dispatch`.
- Проверить возможность регулярного `stplr-spec clean-build` для критичных пакетов, где важна воспроизводимость.
- Рассмотреть `conflicts` для пакетов, которые устанавливают одинаковые публичные бинарники или desktop entries.
- Добавить проверку README на точное совпадение версии, лицензии и архитектур по каждому пакету.
